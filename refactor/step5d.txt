STEP 5D — Cleanup transitional VLAN membership logic and improve developer/user documentation

GOAL

Consolidate the architecture after Step 5A/5B/5C by removing transitional complexity, tightening internal naming and documentation, and significantly improving user-facing Ansible module documentation and examples.

This step is about:
- cleanup
- clarity
- consistency
- documentation
- examples

It is NOT about introducing a new major feature.

WHY THIS STEP EXISTS

After Step 5C, the project has already completed the main canonical architecture migration:

- canonical VLAN membership model exists
- JTCom readback is normalized to canonical
- planning/apply pipeline is driven by canonical desired state
- backend access/native/permit representation exists only at the final conversion boundary

This means the architecture is now strong enough to:
- remove transitional leftovers
- simplify code paths
- improve maintainability
- document the system properly for both developers and users

This step should make the codebase feel intentional, not transitional.

STRICT NON-GOALS

Do NOT:
- redesign the canonical model
- redesign Step 4 policy semantics
- add a new backend/vendor
- redesign public APIs
- add new user-facing syntax
- perform risky behavior changes just for cleanliness

If a cleanup looks risky and not clearly valuable right now, skip it.

PRIMARY THEMES OF THIS STEP

1) Remove transitional / redundant helpers left over from pre-canonical architecture
2) Improve naming consistency around canonical vs backend state
3) Improve docstrings/comments in Python code
4) Improve Ansible module documentation and examples significantly
5) Add a compact architecture overview for future maintainers
6) Keep runtime behavior unchanged

FILES TO MODIFY

Code:

- src/napalm_jtcom/driver.py

Ansible / docs:
