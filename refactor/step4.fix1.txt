PART 2 — ENSURE MODE-NONE FALLBACK DOES NOT BYPASS MODE-CHANGE PROTECTION

CONTEXT

Current Step 4 policy maps desired mode==none to access VLAN 1.
This can implicitly turn:
- trunk -> access

That transition must still be subject to the existing access↔trunk policy.

REQUIRED BEHAVIOR

Given:
- current_per_port
- desired_per_port (after canonical membership ops)
- mode-none fallback to VLAN 1

The final effective desired state after fallback must still be checked against:
- existing access↔trunk mode-change protection

This means:
- if a port was trunk
- and after fallback it becomes access VLAN 1
- then existing mode-change rules still apply:
    - default apply: fail unless allow_port_mode_change=True
    - check mode: warning only

IMPLEMENTATION REQUIREMENTS

1) Review the current planning order in vlan_membership / driver.
2) Ensure fallback to VLAN 1 happens before final mode-change detection,
   OR ensure mode-change detection is rerun after fallback.
3) There must not be any code path where:
   - desired mode becomes none
   - fallback silently rewrites it to VLAN 1
   - trunk→access change slips through without allow_port_mode_change

Preferred implementation:
- compute desired_per_port from canonical membership
- apply mode-none fallback transformation
- then run:
    - mode-change detection
    - changed_ports / changed_vlans / desired_port_vlan
- not the other way around

Required tests:
A) current port is trunk with only tagged VLAN 20, no native
   desired removes tagged 20
   fallback maps to VLAN 1
   result is trunk->access
   default apply must fail unless allow_port_mode_change=True

B) same scenario in check mode
   warning only

C) same scenario with allow_port_mode_change=True
   success

D) access port losing last membership and mapping to VLAN 1
   access->access
   should not require allow_port_mode_change

E) ensure warning type(s) clearly indicate both:
   - fallback occurred
   - mode-change policy applied if relevant

PART 3 — FIX allow_vlan_delete_in_use INTERACTION WITH tagged_set / untagged_set

CONTEXT

When deleting a VLAN still in use, Step 4 currently augments desired canonical input with:
- tagged_remove
- untagged_remove

This can interact badly with set-semantics if the same VlanConfig already contains:
- tagged_set / tagged_ports
- untagged_set / untagged_ports

Potential problems:
- conflicting semantics
- accidental violation of set-vs-add/remove validation
- unclear precedence

REQUIRED POLICY

If a VLAN is marked absent and allow_vlan_delete_in_use=True:
- the policy layer is allowed to detach the VLAN from ports automatically
- this auto-detach must be deterministic and compatible with canonical model validation

RECOMMENDED APPROACH

Do NOT blindly append tagged_remove / untagged_remove onto existing set-based fields.

Instead, for each VLAN marked absent and still in use:

Option A (preferred):
- construct a separate policy-layer effective detach plan that does not mutate the original VlanConfig into an invalid mixed set/remove object
- e.g. augment the effective desired canonical data using fresh VlanConfig instances for deletion policy only

OR

Option B:
- if mutating existing VlanConfig, then:
    - if tagged_set/tagged_ports present, convert to an equivalent effective remove-capable representation first
    - same for untagged_set/untagged_ports
    - ensure final VlanConfig remains internally valid

The simplest safe rule is:
- do not mix *_set/*_ports with *_remove in the same object
- if policy augmentation needs removes, materialize a fresh config object containing only the effective removal operations needed for deletion

REQUIRED BEHAVIOR

Examples:

A) VLAN 20 is absent, currently tagged on ports 5 and 6
- allow_vlan_delete_in_use=True
- effective detach plan removes VLAN 20 from ports 5 and 6 before delete

B) VLAN 20 is absent, currently untagged on port 3
- allow_vlan_delete_in_use=True
- effective detach plan removes untagged VLAN 20 from port 3 before delete

C) Existing desired input includes tagged_set or untagged_set for VLAN 20
- no invalid set/remove mixed config may be produced
- either:
    - conflict clearly, OR
    - cleanly materialize effective removal logic
Recommended: since state=absent dominates, materialize dedicated effective removal logic for delete policy and avoid invalid mixed objects

IMPORTANT

This must not weaken dual syntax conflict detection.
If the user simultaneously says:
- VLAN 20 absent
- and another input path adds membership back to VLAN 20
that remains a hard conflict.

Required tests:
A) VLAN absent + current tagged usage + allow_vlan_delete_in_use=True
   works and does not create invalid mixed set/remove config

B) VLAN absent + current untagged usage + allow_vlan_delete_in_use=True
   works

C) VLAN absent + set-style desired data on same VLAN
   behaves deterministically and validly
   choose and document exact behavior

D) dual syntax conflict still fails even with allow_vlan_delete_in_use=True

E) check mode warning remains accurate

PART 4 — UNIFY WARNING OBJECT STRUCTURE

GOAL

Warnings should be consistently machine-readable.

Current warning objects likely vary by case.
Unify them.

REQUIRED WARNING SCHEMA

Use a consistent structure as much as practical:

{
  "type": "<warning_type>",
  "entity": "port" | "vlan" | "policy",
  "port_id": <int or null>,
  "vlan_id": <int or null>,
  "message": "<human readable message>",
  "hint": "<optional hint>",
  ... case-specific fields ...
}

Minimum required common fields for all warnings:
- type
- entity
- message

Recommended additional common fields:
- port_id (when applicable)
- vlan_id (when applicable)
- hint (when applicable)

CASE-SPECIFIC WARNING TYPES

At minimum normalize:
- untagged_move
- vlan_delete_in_use
- mode_none_mapped_to_vlan1
- existing mode-change warning type (either preserve current name or standardize it too)

EXAMPLES

untagged move:
{
  "type": "untagged_move",
  "entity": "port",
  "port_id": 3,
  "vlan_id": null,
  "message": "Port 3 would move untagged VLAN from 20 to 30.",
  "hint": "Set allow_untagged_move=true to allow this change.",
  "current_untagged_vlan": 20,
  "desired_untagged_vlan": 30
}

delete in use:
{
  "type": "vlan_delete_in_use",
  "entity": "vlan",
  "port_id": null,
  "vlan_id": 20,
  "message": "VLAN 20 is still in use.",
  "hint": "Set allow_vlan_delete_in_use=true to auto-detach and delete it.",
  "affected_ports_tagged": [5, 6],
  "affected_ports_untagged": [3]
}

mode-none fallback:
{
  "type": "mode_none_mapped_to_vlan1",
  "entity": "port",
  "port_id": 5,
  "vlan_id": 1,
  "message": "Port 5 would otherwise have no VLAN membership and was mapped to access VLAN 1.",
  "hint": "",
  "mapped_vlan": 1,
  "reason": "policy_default_vlan1_fallback"
}

- apply/check outputs remain stable and understandable
