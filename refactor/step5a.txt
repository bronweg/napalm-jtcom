STEP 5A — Introduce canonical VLAN membership semantics model as the new single source of truth

GOAL

Introduce a vendor-independent canonical VLAN membership model that becomes the long-term single source of truth for the project.

This step must define the model clearly, add the necessary data structures and conversion helpers, and document the invariants.
It must NOT yet migrate the whole runtime pipeline to use the new model everywhere.

This is a semantic/model step, not a full execution-path migration step.

WHY THIS STEP EXISTS

The current project model is too close to one particular representation at a time:

- some places think in terms of:
  - untagged_vlan
  - tagged_vlans

- the JTCom device readback behaves in terms of:
  - native VLAN
  - permit VLANs (including native VLAN)

Those are not the same thing.

We need a vendor-neutral canonical truth that does not depend on:
- how JTCom CGI encodes trunk state
- how another future switch might encode VLAN state
- how the user chooses to express intent (VLAN-centric or port-centric)

PRIMARY DESIGN DECISION

The new canonical model must describe VLAN membership semantics, not vendor trunk/access encoding.

Canonical truth is:

For a given (port, vlan), the membership state is one of:
- absent
- untagged
- tagged

This means:
- vendor-specific concepts like native_vlan / permit_vlans are NOT canonical truth
- they are backend/device representation
- access/trunk mode is also not the deepest truth; it is derived or backend-facing

IMPORTANT

Do NOT throw away the existing VlanConfig add/remove/set model in this step.
Do NOT try to rewrite the whole Step 2 engine in this step.

The objective is:
- define the new canonical state model
- add conversion helpers
- add tests
- make the project ready for Step 5B and 5C

STRICT NON-GOALS

Do NOT:
- rewrite full apply path
- rewrite full verification path
- rewrite full policy layer
- remove existing VLAN-centric or port-centric input
- change Step 4 behavior
- change Ansible API
- change vendor payload code yet
- attempt a giant end-to-end migration in one PR

This step is foundational only.

TARGET MODEL

Introduce a canonical per-port membership state model that expresses wire-level VLAN semantics.

At minimum, the model must be able to represent:

- for each port:
    - exactly zero or one untagged VLAN
    - zero or more tagged VLANs

This can be represented either as:
A) a port-level structure:
   - untagged_vlan: int | None
   - tagged_vlans: set[int]

OR
B) a more explicit per-(port,vlan) state structure internally:
   - membership state enum: absent | untagged | tagged

Recommendation:
Use the existing practical port-level shape as the first implementation:
- untagged_vlan
- tagged_vlans

BUT formalize its meaning strictly as:
- untagged_vlan = VLAN transmitted untagged on wire
- tagged_vlans = VLANs transmitted tagged on wire

This is the key distinction:
- tagged_vlans is NOT permit_vlans
- permit_vlans is a backend/device representation that may be derived from canonical state

NEW SEMANTIC RULES

Canonical model rules:
1) A port may have at most one untagged VLAN
2) A VLAN cannot be both untagged and tagged on the same port in canonical form
3) Tagged VLANs represent on-wire tagged membership only
4) Native VLAN / permit VLAN concepts are device-model concepts and must not leak into canonical state
5) Access/trunk mode may be derived from canonical state:
   - access if one untagged VLAN and no tagged VLANs
   - trunk if one untagged VLAN and one or more tagged VLANs, or if backend supports tagged-only trunks
   - other backend-specific forms may exist later
6) Canonical model must remain vendor-independent

PART 1 — INTRODUCE CANONICAL STATE TYPES

Define explicit types/structures for canonical membership semantics.

Minimum recommended structures:

A) PortMembershipState
This may already exist, but its meaning must be clarified and formalized.

If kept, its semantics must be documented as:
- untagged_vlan: VLAN transmitted untagged on wire
- tagged_vlans: VLANs transmitted tagged on wire

B) Optional helper TypedDict / dataclass for backend-oriented port VLAN representation

Example:
- PortBackendVlanState or PortTrunkAccessState

Suggested fields:
- mode: "access" | "trunk"
- access_vlan: int | None
- native_vlan: int | None
- permit_vlans: list[int]

Purpose:
This structure is NOT canonical truth.
It is the backend/device-side representation for devices like JTCom.

C) Optional enum-like documentation for per-(port,vlan) membership semantics
Even if not implemented as an actual Enum everywhere, document clearly:
- absent
- untagged
- tagged

Acceptance:
- the project now clearly distinguishes canonical state from backend representation

PART 2 — FORMALIZE INVARIANTS IN CODE COMMENTS / DOCSTRINGS

In relevant files/docstrings, explicitly document:

Canonical semantics:
- untagged_vlan and tagged_vlans describe on-wire tagging behavior
- native/permit is not canonical truth
- canonical state must not contain the same VLAN both untagged and tagged

Backend semantics:
- on JTCom trunk ports, permit_vlans includes native_vlan
- therefore backend readback must be normalized to canonical state
- therefore canonical tagged_vlans must not blindly mirror permit_vlans

This documentation is important.
Future maintainers must not repeat the current misunderstanding.

Acceptance:
- any old wording implying tagged_vlans == permit_vlans must be corrected

PART 3 — ADD EXPLICIT CONVERSION HELPERS

This is the key functional part of 5A.

Implement explicit helpers for conversion between:

A) Canonical state -> JTCom backend representation

Suggested helper:
- canonical_to_jtcom_port_vlan_state(state: PortMembershipState) -> BackendState

Rules:
1) If canonical state is:
   - untagged_vlan = X
   - tagged_vlans = empty
   then backend state is access:
   - mode = access
   - access_vlan = X
   - native_vlan = None
   - permit_vlans = []

2) If canonical state is:
   - untagged_vlan = X
   - tagged_vlans = {A, B, ...}
   then backend state is trunk:
   - mode = trunk
   - access_vlan = None
   - native_vlan = X
   - permit_vlans = sorted({X} union tagged_vlans)

3) If canonical state is:
   - untagged_vlan = None
   - tagged_vlans non-empty
   decide what to do in this step:
   Recommendation:
   - keep support decision explicit and documented
   - if current JTCom backend cannot express this safely, conversion helper should raise a clear ValueError
   - do NOT silently invent native VLAN here

4) If canonical state is empty:
   - untagged_vlan = None
   - tagged_vlans = empty
   do NOT decide final policy here
   Step 4 policy already maps none to VLAN 1 later in runtime.
   In this model step, the helper may:
   - either raise ValueError for JTCom conversion
   - or leave this case unhandled and document that policy layer must resolve it before backend conversion

B) JTCom backend representation -> canonical state

Suggested helper:
- jtcom_to_canonical_port_vlan_state(backend_state) -> PortMembershipState

Rules:
1) Access mode:
   - access_vlan = X
   -> canonical:
      untagged_vlan = X
      tagged_vlans = empty

2) Trunk mode:
   - native_vlan = X
   - permit_vlans = [X, A, B, ...]
   -> canonical:
      untagged_vlan = X
      tagged_vlans = {A, B, ...}
   That is:
      tagged_vlans = set(permit_vlans) - {native_vlan}

3) Backend trunk invariant:
   - native_vlan must be included in permit_vlans
   If not:
   - raise a clear ValueError or backend parse error

These two conversion helpers are the foundation of the whole future refactor.

Acceptance:
- conversions are deterministic
- tests prove round-trip semantics for supported cases

PART 5 — TESTS

Add dedicated tests for the new canonical model and conversions.

Required cases:

A) Canonical -> JTCom access
input canonical:
- untagged_vlan=10
- tagged_vlans=set()
expected backend:
- mode=access
- access_vlan=10
- native_vlan=None
- permit_vlans=[]

B) Canonical -> JTCom trunk
input canonical:
- untagged_vlan=1
- tagged_vlans={61}
expected backend:
- mode=trunk
- native_vlan=1
- permit_vlans=[1, 61]

C) JTCom -> Canonical access
input backend:
- mode=access
- access_vlan=10
expected canonical:
- untagged_vlan=10
- tagged_vlans=set()

D) JTCom -> Canonical trunk
input backend:
- mode=trunk
- native_vlan=1
- permit_vlans=[1, 61]
expected canonical:
- untagged_vlan=1
- tagged_vlans={61}

E) Round-trip trunk
canonical:
- untagged_vlan=10
- tagged_vlans={20, 30}
canonical_to_jtcom -> jtcom_to_canonical
expected:
- same canonical state

F) JTCom invalid trunk representation
input:
- mode=trunk
- native_vlan=10
- permit_vlans=[20, 30]
expected:
- ValueError because native_vlan missing from permit_vlans

G) Unsupported tagged-only canonical state (if not supported yet)
input canonical:
- untagged_vlan=None
- tagged_vlans={20}
expected:
- explicit ValueError from JTCom conversion helper
unless you intentionally support it with a well-defined rule

H) Empty canonical state
input canonical:
- untagged_vlan=None
- tagged_vlans=set()
expected:
- documented conversion behavior
- preferably ValueError for JTCom conversion helper at this stage

I) Canonical invariant
same VLAN cannot be both untagged and tagged
if a constructor/helper can create such invalid state, test that normalization or validation rejects it
