STEP 3 — Introduce port-centric VLAN membership input (dual syntax) and merge it into the canonical membership model

GOAL

Extend the system so that VLAN membership can be expressed from both sides:

1) VLAN-centric input (already supported)
2) Port-centric input (new)

Both syntaxes must be normalized into the same canonical VLAN membership model introduced in Step 1 and executed by the Step 2 membership engine.

This step is about INPUT NORMALIZATION and CONFLICT DETECTION.
It must not redesign the low-level runtime engine.

TARGET OUTCOME

After this step, users must be able to express intent in either of these forms:

VLAN-centric:
- vlans:
    20:
      tagged_add: [5]
    30:
      untagged_add: [3]

Port-centric:
- ports:
    5:
      trunk_add_vlans: [20]
    3:
      access_vlan: 30

Both must be translated into one canonical VLAN membership patch before planning/apply.

STRICT NON-GOALS

Do NOT:
- redesign Step 2 membership engine
- change device write path
- add new switch CGI endpoints
- redesign Ansible collection structure
- implement VLAN deletion policy changes
- implement advanced policy layer (fail_if_in_use, auto_detach, etc.)
- support every imaginable port UX variant

This step is specifically about:
- port-centric VLAN membership input
- dual syntax merge
- conflict detection
- normalization into existing canonical model

HIGH-LEVEL DESIGN

Current architecture:
- user input
- VlanConfig canonical membership ops
- membership engine
- diff/apply

New architecture after Step 3:
- user input
  -> VLAN-centric input
  -> Port-centric VLAN input
  -> normalize both into canonical VLAN membership ops
  -> merge
  -> fail on conflicts
  -> existing Step 2 membership engine
  -> diff/apply

PRIMARY DESIGN RULE

Canonical source of truth remains VLAN-centric membership operations.

That means:
- internal apply engine still consumes VlanConfig membership operations
- port-centric input is only an alternative syntax
- port-centric input must be translated into VLAN membership ops before planning

==================================================
2) DEFINE PORT-CENTRIC MEMBERSHIP SEMANTICS
==================================================

This is the most important specification part.

For one port:

A) access_vlan
- means: set this port as untagged member of VLAN X
- translated canonical effect:
    VLAN X -> untagged_add [port]
- and if current/other input says another untagged VLAN applies to this port,
  conflict detection or overwrite rules will decide later

B) native_vlan
- means: on a trunk/hybrid port, incoming untagged frames belong to VLAN X
- translated canonical effect:
    VLAN X -> untagged_add [port]

C) trunk_add_vlans
- means: add these VLANs as tagged on this port
- translated canonical effect:
    for each VLAN in list:
        VLAN -> tagged_add [port]

D) trunk_remove_vlans
- means: remove these VLANs from tagged membership on this port
- translated canonical effect:
    for each VLAN in list:
        VLAN -> tagged_remove [port]

E) trunk_set_vlans
- means: replace the full tagged VLAN set for this port
- translated canonical effect:
    this cannot be translated as independent add/remove per VLAN only;
    it must be expressed as:
      - for all currently tagged VLANs on this port, remove this port
      - for all desired tagged VLANs, add this port
  Because current state matters, this translation must happen at runtime/planning stage,
  not as a static pure model conversion.

Therefore:
- access_vlan/native_vlan can be converted statically to untagged_add semantics
- trunk_add_vlans / trunk_remove_vlans can be converted statically
- trunk_set_vlans requires current runtime state and must be handled in a planner helper

==================================================
4) MERGE VLAN-CENTRIC AND PORT-CENTRIC INPUTS
==================================================

Implement a merge layer that combines:
- explicit VLAN-centric desired_vlans
- translated port-centric membership operations

This layer must produce:
- final canonical desired_vlans input for the Step 2 engine

Key rule:
- merge to canonical
- do not give one syntax silent priority over the other

Algorithm:
1. Start from original desired_vlans
2. Convert port-centric input into canonical VLAN membership operations
3. Merge into a new combined desired_vlans map
4. Detect conflicts
5. Return combined map

==================================================
5) CONFLICT DETECTION RULES
==================================================

This is mandatory.
If both syntaxes describe incompatible intent, fail explicitly.

Examples of conflicts:

A) Two different untagged VLANs for the same port
Example:
- VLAN-centric:
    VLAN 20 -> untagged_add [3]
- Port-centric:
    port 3 -> access_vlan 30
Result:
- conflict
- raise explicit error

B) Contradictory tagged instructions
Example:
- VLAN-centric:
    VLAN 20 -> tagged_add [5]
- Port-centric:
    port 5 -> trunk_remove_vlans [20]
Result:
- conflict
- raise explicit error

C) trunk_set_vlans conflicting with VLAN-centric adds/removes
Example:
- Port-centric:
    port 5 -> trunk_set_vlans [10,20]
- VLAN-centric:
    VLAN 30 -> tagged_add [5]
Result:
- conflict unless merge result is provably identical
- prefer explicit fail in this step

Conflict policy for Step 3:
- explicit conflict -> ValueError (or dedicated DualSyntaxConflictError)
- do not silently choose one syntax over the other

Recommended custom exception:
- DualSyntaxConflictError(ValueError)

Error message should include:
- port_id and/or vlan_id
- both conflicting intents
- enough detail to understand the issue

==================================================
6) HANDLE PORT-CENTRIC INPUT WITH NO CONFLICTS
==================================================

If both syntaxes are present and compatible:
- merge them

Examples:

A) Compatible
- VLAN-centric:
    VLAN 20 -> tagged_add [5]
- Port-centric:
    port 5 -> native_vlan 10
Result:
- port 5 trunk with native 10 and tagged 20
- no conflict

B) Compatible
- VLAN-centric:
    VLAN 20 -> tagged_add [5]
- Port-centric:
    port 5 -> trunk_add_vlans [30]
Result:
- final canonical:
    VLAN 20 -> tagged_add [5]
    VLAN 30 -> tagged_add [5]
