TASK: Step 5C follow-up hardening and cleanup

GOAL

Harden the Step 5C migration so that the architectural boundary between:

- canonical VLAN membership state
and
- JTCom backend access/native/permit representation

is fully enforced, explicit, test-covered, and resistant to future regressions.

This is not a redesign of Step 5C.
This is a correctness + hardening pass.

PRIMARY OBJECTIVES

1) Ensure there is exactly one canonical -> JTCom compilation path
2) Ensure unsupported canonical states fail explicitly at backend compilation boundary
3) Ensure verification expected-state remains purely canonical and is never backend-shaped
4) Ensure Step 4 policy logic operates only on canonical state
5) Protect desired_per_port from accidental mutation during apply/verification flow
6) Make the canonical/backend boundary obvious in code comments
7) Add targeted tests to lock the architecture in place

STRICT NON-GOALS

Do NOT:
- redesign canonical model
- redesign Step 4 policy behavior
- redesign Ansible API
- redesign JTCom backend model
- start Step 5D
- do broad unrelated cleanup

This task is a hardening pass for Step 5C only.

- src/napalm_jtcom/utils/vlan_membership.py

PART 1 — ENFORCE A SINGLE CANONICAL -> JTCom COMPILATION PATH

Problem:
There must be exactly one authoritative place where canonical port state is converted into JTCom backend representation.

Required rule:
The only supported compiler from canonical state to JTCom backend state is:

- canonical_to_jtcom_port_vlan_state(...)

No other code path may manually construct:
- access_vlan
- native_vlan
- permit_vlans

from canonical state.

Required work:
1) Do a repo-wide grep for:
   - access_vlan
   - native_vlan
   - permit_vlans
   especially in:
   - src/napalm_jtcom/utils/*
   - src/napalm_jtcom/driver.py
   - src/napalm_jtcom/client/*
2) Inspect all occurrences and classify them:
   A) backend representation consumption/transport -> allowed
   B) canonical->backend construction outside the shared helper -> forbidden
3) Remove any remaining ad hoc reconstruction logic outside canonical_to_jtcom_port_vlan_state(...)

Acceptance:
- there is no code outside the shared conversion helper that manually assembles permit/native/access from canonical desired state
- backend write boundary is singular and obvious

Required tests:
- at least one regression test that would fail if someone later manually diverged from canonical_to_jtcom_port_vlan_state(...)
- if direct mocking is practical, assert that apply path uses the conversion helper output rather than manual reconstruction

PART 2 — EXPLICITLY FAIL ON UNSUPPORTED CANONICAL STATES AT APPLY BOUNDARY

Problem:
Canonical state can validly represent cases that JTCom cannot compile:
- tagged-only canonical state
- empty canonical state

That failure must happen:
- explicitly
- clearly
- at backend compilation boundary
and not through accidental downstream behavior.

Required behavior:
In _apply_vlan_membership_plan(), when compiling canonical desired_per_port to JTCom state:
- call canonical_to_jtcom_port_vlan_state(...)
- catch ValueError from unsupported JTCom compilation cases
- re-raise a clearer backend-boundary exception or ValueError with context including port_id

Recommended behavior:
Wrap with something like:

- "Port 5 canonical state cannot be compiled to JTCom backend: <reason>"

Do NOT:
- silently coerce
- silently skip
- rely on downstream write helpers to fail later in a vague way

Required tests:
A) tagged-only canonical state reaches apply boundary
   -> fails explicitly with clear message mentioning backend limitation
B) empty canonical state reaches apply boundary
   -> fails explicitly with clear message mentioning policy/backend contract
C) ensure these failures are not misreported as verification failures

Required tests:

PART 5 — PROTECT desired_per_port FROM ACCIDENTAL MUTATION

Problem:
desired_per_port is now the primary runtime desired model.
If it is mutated in-place during apply or verification flow, subtle bugs can appear.

Required decision:
Use one of these approaches:

Preferred approach:
- make safe copies at mutation boundaries (deepcopy where needed)

Alternative approach:
- make canonical state structures immutable if low-risk and practical
  (probably too invasive for this follow-up unless already easy)

Recommended minimum fix:
1) Identify all places where desired_per_port or nested port states may be mutated
2) If apply path needs mutation-like transformations, do them on copies
3) Ensure verification reads stable canonical desired state, not a mutated artifact

Acceptance:
- desired_per_port used for verification expectations cannot be accidentally altered by backend compilation/apply flow

Required tests:
A) a test that planning result remains unchanged after _apply_vlan_membership_plan() preparation steps
B) if direct object identity is easy to observe, verify that backend compilation works from copies, not by mutating canonical plan in place
