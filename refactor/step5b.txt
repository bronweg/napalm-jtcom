STEP 5B — Move JTCom readback and post-apply verification to the canonical VLAN membership model

GOAL

Migrate device readback interpretation and post-apply verification so they operate on the canonical VLAN membership model introduced in Step 5A.

This step must fix the current false verification failures caused by comparing:
- canonical desired state
against
- JTCom-style backend readback semantics

The canonical model must remain the single source of truth for comparison.

WHY THIS STEP EXISTS

Current problem:
- desired state is interpreted canonically
- JTCom readback returns backend/device-oriented trunk information
- on JTCom, trunk readback effectively behaves as:
    native_vlan + permit_vlans
  where permit_vlans includes native_vlan

If this backend representation is compared directly to canonical state, verification fails incorrectly.

Example:
Desired canonical state:
- untagged_vlan = 1
- tagged_vlans = {61}

JTCom readback:
- native_vlan = 1
- permit_vlans = [1, 61]

Correct canonical interpretation of that readback is:
- untagged_vlan = 1
- tagged_vlans = {61}

This step must make that normalization explicit and authoritative.

PRIMARY OBJECTIVE

All readback and verification paths must compare:
- canonical expected state
to
- canonical observed state

NOT:
- canonical expected state
to
- raw or semi-raw JTCom backend readback representation

STRICT NON-GOALS

Do NOT:
- migrate full planning/apply pipeline yet
- redesign Step 4 policy logic
- redesign user-facing API
- redesign Ansible interface
- rewrite port-centric / VLAN-centric input compilation
- attempt Step 5C in this PR

This step is specifically about:
- readback normalization
- post-apply verification
- current-state modeling for comparison

FILES TO MODIFY

Primary:
- src/napalm_jtcom/utils/vlan_membership.py
- src/napalm_jtcom/driver.py

Likely:
- src/napalm_jtcom/parser/vlan.py

- maybe helper modules related to current_per_port construction

Tests:

- tests/unit/test_device_plan.py
- tests/unit/test_vlan_diff.py

- add new dedicated tests if needed, for example:
  - tests/unit/test_canonical_readback_verification.py

HIGH-LEVEL TARGET STATE

After this step:

1) There is a clear JTCom backend readback shape
2) That backend shape is converted to canonical port membership state
3) current_per_port used by verification is canonical
4) post-apply verification compares canonical-to-canonical
5) The known false failure case with native VLAN included in permit_vlans disappears

PART 1 — DEFINE / IDENTIFY JTCom BACKEND READBACK REPRESENTATION

Before changing behavior, make sure the code has an explicit concept of JTCom backend readback state.

The project must distinguish:

A) Canonical port state
- untagged_vlan
- tagged_vlans
- on-wire semantics only

B) JTCom backend port VLAN state
- access_vlan
- or native_vlan + permit_vlans
- device semantics

Required action:
- inspect current readback code path
- identify where backend readback is currently flattened too early or interpreted ambiguously
- make that representation explicit where helpful

If useful, introduce a typed helper structure for JTCom readback state.
If not needed, keep it lightweight, but the distinction must be clear in code.

PART 2 — MAKE build_current_per_port_from_vlans() CANONICAL

This is likely the most important functional change.

Current helper(s) that build current_per_port must produce canonical state, not semi-backend state.

If current code currently interprets a JTCom trunk port as:
- untagged_vlan = native
- tagged_vlans = permit_vlans
then that is wrong for canonical semantics.

Required behavior:
For JTCom trunk readback:
- native_vlan = X
- permit_vlans = [X, A, B, ...]
must produce canonical:
- untagged_vlan = X
- tagged_vlans = {A, B, ...}

That is:
- tagged_vlans = set(permit_vlans) - {native_vlan}

Required implementation:
- find the readback/build-current-state path
- ensure it uses jtcom_to_canonical_port_vlan_state(...) or equivalent shared conversion helper from Step 5A
- avoid copy-pasting conversion logic if possible

Acceptance requirement:
- current_per_port everywhere used for planning/verification must be canonical

PART 3 — REUSE STEP 5A CONVERSION HELPERS

Very important:
Do not duplicate the conversion logic again.

Use the Step 5A helpers:
- canonical_to_jtcom_port_vlan_state(...)
- jtcom_to_canonical_port_vlan_state(...)

In this step, the second helper is especially important.

If current code does not yet naturally route through these helpers, refactor enough so that:
- JTCom readback normalization reuses the canonical conversion helper
- future bugs cannot arise from multiple slightly different implementations

Acceptance:
- there should be one authoritative JTCom -> canonical conversion path

PART 4 — FIX POST-APPLY VERIFICATION TO COMPARE CANONICAL STATES

Current post-apply verification must be updated so that:

expected:
- canonical desired state for changed ports

actual:
- canonical state rebuilt from device readback

This means:
- no direct comparison against permit-vlan semantics
- no raw trunk/native payload comparison unless explicitly converted first

Required implementation:
1) After apply, read device state
2) Convert device readback to canonical current_per_port
3) Compare against canonical expected changed-port state
4) Remaining diff must be expressed in canonical terms

Important:
Verification should remain strict.
Do NOT “weaken verification”.
The fix is:
- compare the correct normalized model
not:
- ignore differences

Acceptance:
- the previous false mismatch case should disappear
- real mismatches should still fail
