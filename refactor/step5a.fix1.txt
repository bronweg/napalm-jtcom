==================================================
2.1) canonical_to_jtcom: allow empty canonical state as a valid canonical concept,
     but explicitly document that JTCom conversion requires policy resolution first
==================================================

Current issue:
The helper currently raises ValueError for empty canonical state.

That is acceptable in practice for JTCom conversion, but the reason must be documented correctly:
- empty canonical state is not invalid canonical truth
- it is simply not directly convertible to JTCom backend representation unless policy layer resolves it first

Required change:
- Keep the helper behavior of refusing direct JTCom conversion for empty canonical state
- BUT make the contract explicit in code and docstring

Preferred implementation:
- keep raising ValueError for JTCom conversion
- but use a message like:

  "Empty canonical port state cannot be converted directly to JTCom backend state; policy layer must resolve it first."

Required docstring addition:
- explain that canonical empty state is allowed as a canonical concept
- explain that JTCom conversion expects the policy layer to resolve it before backend compilation

Required tests:
- keep existing test or adjust it so it checks the more precise message
