STEP 1 — FIXUP TASK
Correct the Step 1 VLAN membership model implementation so that it fully matches the intended semantics,
especially around legacy field handling, explicit empty-list replacement, and validation.

CONTEXT

The current Step 1 implementation is close, but it has several semantic bugs:

1) Legacy fields `tagged_ports` and `untagged_ports` currently use `default_factory=list`,
   which makes it impossible to distinguish:
   - field not provided
   - field explicitly provided as an empty list

   This breaks the intended semantics:
   - legacy fields must behave as REPLACE (`set`)
   - therefore `tagged_ports=[]` must mean `tagged.set = set()`
   - not “no-op”

2) Several conflict checks currently rely on truthiness instead of explicit field presence.
   This means empty lists are silently ignored and do not participate in conflict detection.

3) Runtime validation for `state` must be explicit, not only type-hinted.

4) Minor cleanup is needed in helper functions, imports, and formatting.

GOAL

Fix the model implementation in `src/napalm_jtcom/model/vlan.py` so that:

- legacy fields can express “unset” vs “explicit empty”
- conflict detection is based on field presence, not truthiness
- normalized_membership() correctly treats empty legacy lists as `set()`
- state is explicitly runtime-validated
- helper behavior and error messages are improved
- all tests are updated and extended accordingly

STRICT NON-GOALS

Do NOT:
- change driver behavior
- change device logic
- change action plugin behavior
- change Ansible input schema yet
- change any switch-side behavior
- start Step 2 logic

This is still model-only.

REQUIRED CHANGES

==================================================
1) CHANGE LEGACY FIELDS TO OPTIONAL
==================================================

In `VlanConfig`, change:

- tagged_ports: list[int] = field(default_factory=list)
- untagged_ports: list[int] = field(default_factory=list)

to:

- tagged_ports: list[int] | None = None
- untagged_ports: list[int] | None = None

RATIONALE:
- None = field not provided
- [] = field explicitly provided and means replace with empty set

This distinction is mandatory for correct semantics.

Do NOT change the new operation fields:
- tagged_add / tagged_remove / tagged_set
- untagged_add / untagged_remove / untagged_set
They may remain optional with default None.

2) UPDATE DOCSTRINGS / FIELD COMMENTS

Update the class docstring (and/or field comments if present) to clearly state:

- `tagged_ports` and `untagged_ports` are legacy backward-compatible fields
- when provided, they mean full replacement semantics
- explicit empty list means replace with empty set
- None means field omitted / unspecified

Also document the new operation fields:
- add = additive patch
- remove = subtractive patch
- set = explicit replacement

3) FIX VALIDATION TO USE FIELD PRESENCE, NOT TRUTHINESS
==================================================

Update conflict validation in `__post_init__`.

Current buggy patterns are like:
- if self.tagged_ports and ...
- if self.untagged_ports and ...
- if self.tagged_set is not None and (self.tagged_add or self.tagged_remove)
- etc.

These must be rewritten using explicit “field is present” semantics.

Required rules:

A) Tagged side
- if tagged_set is not None, then tagged_add must be None and tagged_remove must be None
- if tagged_ports is not None, then tagged_add must be None, tagged_remove must be None, tagged_set must be None

B) Untagged side
- if untagged_set is not None, then untagged_add must be None and untagged_remove must be None
- if untagged_ports is not None, then untagged_add must be None, untagged_remove must be None, untagged_set must be None

IMPORTANT:
Use `is not None`, not truthiness.

Examples that MUST now fail:
- tagged_ports=[] with tagged_set=[1]
- tagged_ports=[] with tagged_add=[1]
- untagged_ports=[] with untagged_remove=[1]
- tagged_set=[] with tagged_add=[]
- untagged_set=[] with untagged_remove=[]

If the conflicting field is present at all, it is a conflict, regardless of emptiness.

8) FIX normalized_membership() SEMANTICS
==================================================

This is the most important functional fix.

The method must now behave as follows:

Return shape:

{
  "tagged": {
    "add": set[int],
    "remove": set[int],
    "set": set[int] | None,
  },
  "untagged": {
    "add": set[int],
    "remove": set[int],
    "set": set[int] | None,
  },
}

A) Tagged side
- if self.tagged_set is not None:
    tagged.set = set(self.tagged_set)
    tagged.add = empty set
    tagged.remove = empty set

- elif self.tagged_ports is not None:
    tagged.set = set(self.tagged_ports)
    tagged.add = empty set
    tagged.remove = empty set

- else:
    tagged.set = None
    tagged.add = normalized tagged_add
    tagged.remove = normalized tagged_remove

B) Untagged side
- if self.untagged_set is not None:
    untagged.set = set(self.untagged_set)
    untagged.add = empty set
    untagged.remove = empty set

- elif self.untagged_ports is not None:
    untagged.set = set(self.untagged_ports)
    untagged.add = empty set
    untagged.remove = empty set

- else:
    untagged.set = None
    untagged.add = normalized untagged_add
    untagged.remove = normalized untagged_remove

CRITICAL:
- Explicit empty legacy list must produce set()
- Explicit empty tagged_set / untagged_set must produce set()
- Only omitted field produces set=None
