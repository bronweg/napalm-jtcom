STEP 2 — FIXUP TASK
Correct the remaining semantic and safety issues in the Step 2 VLAN membership runtime implementation.

CONTEXT

Step 2 implementation is already strong:
- canonical membership operations are wired into runtime
- add/remove/set semantics exist
- full per-port desired trunk permit lists are computed
- access↔trunk changes are guarded by allow_port_mode_change
- check mode returns warnings instead of failing
- post-apply verification exists
- Ansible accidental replace bug was fixed

However, there are still several important issues that must be corrected before Step 2 can be considered safe to merge.

PRIMARY PROBLEMS TO FIX

1) device_diff currently destroys the semantic distinction between:
   - None  -> field omitted / unspecified
   - []    -> explicit empty set / replacement
   using patterns like:
   - set(current.tagged_ports or [])
   - set(current.untagged_ports or [])

2) _new_vlan_side_membership() produces misleading output for remove-only operations,
   effectively collapsing "remove" into "empty set" semantics for diff purposes.

3) Runtime policy for desired port mode == "none" is currently hard-fail.
   This may block valid workflows and needs to be formalized explicitly.
   Either:
   - keep fail-fast, but make it fully intentional, documented, and check-mode aware
   OR
   - support it safely in a deterministic way.
   For this fixup, keep the existing fail-fast behavior unless you are certain the device API supports a safe "none" mode.
   But improve how it is handled and surfaced.

GOAL

Fix the remaining Step 2 issues without redesigning the architecture.

STRICT NON-GOALS

Do NOT:
- redesign the public API
- add dual syntax from port-centric input
- add new flags beyond what already exists, unless absolutely required
- change collection structure
- change Step 1 model semantics
- implement deletion policy redesign
- introduce a new "none" switch-side mode unless proven supported

1) FIX device_diff TO PRESERVE None vs [] SEMANTICS
==================================================

File:
- src/napalm_jtcom/utils/device_diff.py

Problem:
The current code uses:
- set(current.tagged_ports or [])
- set(current.untagged_ports or [])

This collapses:
- None -> empty set

This is not allowed, because Step 1 explicitly established:
- None = unspecified
- []   = explicit replacement with empty set

Requirements:
A) Stop using `or []` for membership fields that are semantically nullable.
B) Preserve the distinction in diff computation.

Implement explicit handling:

For current.tagged_ports:
- if current.tagged_ports is None:
    current_tagged = None
- else:
    current_tagged = set(current.tagged_ports)

For current.untagged_ports:
- if current.untagged_ports is None:
    current_untagged = None
- else:
    current_untagged = set(current.untagged_ports)

Then update diff logic accordingly.

Semantic rules for diff:
- if desired config omits a side entirely (no set/add/remove on that side):
    that side should not appear as a diff just because current is None or []
- if desired config explicitly sets a side:
    compare against current accurately
- if desired config does additive/subtractive changes:
    compute the resulting set from current if current is known
- if current is None and desired uses add/remove without explicit set:
    decide one consistent behavior and document it

Required behavior for this task:
- treat current None as "unknown/unspecified baseline"
- but since Step 2 runtime already reads concrete current membership from device, the diff helper should preferably operate on concrete runtime entry data whenever possible
- do not fabricate [] from None

Implementation guidance:
- if this helper is intended only for runtime diff against actual switch state, current should usually be concrete from VlanEntry
- if current can still be model-level nullable, then preserve that nullability and do not coerce

Acceptance requirement:
- No code path in device_diff may use `or []` on semantically nullable membership fields.

2) FIX _new_vlan_side_membership() / MEMBERSHIP DIFF UX
==================================================

File:
- src/napalm_jtcom/utils/device_diff.py

Problem:
Current implementation:

def _new_vlan_side_membership(cfg, side):
    ...
    if explicit_set is not None:
        return set(explicit_set)
    return set(membership["add"] or set())

This silently ignores:
- remove-only operations
- mixed add/remove operations

This makes diff misleading.

Required fix:
Do NOT use _new_vlan_side_membership() to represent resulting membership for diff.

Instead, compute resulting membership using the same logic as runtime semantics:
- start from current side membership
- apply set if present
- otherwise apply add/remove
- remove overlap with untagged where required

In other words:
- diff must show actual resulting membership
- not only "set" or "add"

Recommended approach:
- remove _new_vlan_side_membership() entirely
OR
- replace it with a helper that accepts both current and desired and computes final resulting membership

You already have similar logic in:
- _vlan_membership_diffs()
Make this the single source of truth for membership diff rendering.

Acceptance requirement:
- remove-only membership changes must be represented correctly in diff
- add+remove combination must be represented correctly in diff
- no helper should collapse remove-only semantics into "empty set"

3) FORMALIZE AND IMPROVE "mode == none" BEHAVIOR
==================================================

File:
- src/napalm_jtcom/driver.py
- tests/unit/test_vlan_membership_apply.py

Current behavior:
In _apply_vlan_membership_plan(), if desired per-port mode becomes "none", the code raises:

- ValueError("desired VLAN mode 'none' cannot be applied safely; JTCom CGI exposes only access/trunk VLAN port modes")

This is acceptable as a fail-safe approach, but it must be made fully consistent.

Required behavior:
A) Keep fail-fast by default for real apply mode.
B) In check mode, do not hard-fail for mode == none.
   Instead:
   - include a warning
   - include the before/after desired membership in diff/metadata

Rationale:
- check mode should be useful for planning
- apply mode should remain safe unless/until there is proven support for "none"

Implementation requirements:
1. Detect desired mode == "none" during planning phase, not only during low-level apply.
2. Add a warning message for such ports in check mode.
3. In real apply mode:
   - raise a clear exception before partial application begins if any changed port would end up in mode == "none"
   - do not wait until halfway through applying ports

Recommended message structure:
- include port_id
- include desired state summary
- explain that JTCom CGI supports only access/trunk
- explain that this operation currently cannot be applied safely

Important:
- This preflight check must happen before any device mutation.
- Avoid partial application followed by ValueError.

Acceptance requirement:
- check_mode with desired mode == none returns warning, not exception
- apply mode with desired mode == none fails before mutating device
- tests cover both paths
