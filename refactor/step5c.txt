STEP 5C — Move planning and apply pipeline to the canonical VLAN membership model

GOAL

Make the canonical VLAN membership model introduced in Step 5A and adopted by readback/verification in Step 5B the central runtime model for planning and apply.

After this step:
- desired state planning must operate in canonical terms
- policy layer must operate on canonical state
- backend/device-specific access/native/permit representation must be treated as a compilation target only
- apply path must compile canonical state into JTCom backend representation at the last possible stage

This is the key architectural migration step.

WHY THIS STEP EXISTS

Current situation after Step 5B:
- canonical model exists
- JTCom readback is normalized into canonical model
- verification compares canonical-to-canonical

But planning/apply logic may still contain assumptions shaped by older intermediate representations or by JTCom-like backend semantics.

This step completes the architectural pivot:

    canonical state becomes the real center of runtime planning

while:
    JTCom access/native/permit becomes a backend adapter representation

STRICT NON-GOALS

Do NOT:
- redesign user-facing APIs
- remove VLAN-centric input
- remove port-centric input
- redesign Step 4 policies
- add support for new device types
- perform broad documentation rewrite beyond what is needed
- aggressively delete every old helper in one go if it increases risk

This step is about:
- moving the planning/apply pipeline to canonical state
- minimizing backend leakage
- keeping behavior the same while making the architecture cleaner

HIGH-LEVEL TARGET STATE

After Step 5C, the effective runtime flow should be conceptually:

1) User input
   - VLAN-centric syntax
   - port-centric syntax

2) Normalize and merge into canonical desired membership state

3) Apply Step 4 policy layer on canonical state

4) Compare canonical desired state against canonical current state

5) Produce changed ports / canonical desired port states

6) Only at the final backend boundary:
   compile canonical port state into JTCom representation:
   - access_vlan
   - or native_vlan + permit_vlans

7) Send to device

8) Read back from device and canonicalize (already Step 5B)

9) Verify canonical expected vs canonical actual

PART 1 — DEFINE WHAT “CANONICAL DESIRED STATE” MEANS IN RUNTIME

You now need a concrete runtime structure representing the desired per-port canonical state.

Recommended shape:
- reuse the same canonical PortMembershipState used in Step 5A/5B:
    - untagged_vlan
    - tagged_vlans
with Step 5A semantics:
    - untagged = on-wire untagged VLAN
    - tagged = on-wire tagged VLANs

This must become the primary runtime desired state for planning/apply.

Important:
- JTCom backend representation must no longer be the primary desired model during planning
- backend representation should be derived from canonical desired state near the write boundary

Acceptance:
- there is a clearly identifiable canonical desired_per_port structure used in planning

PART 2 — BUILD DESIRED CANONICAL STATE FROM USER INPUT

Current inputs still come in two forms:
- VLAN-centric
- port-centric

Those should continue to be supported.

Required behavior:
A) VLAN-centric input must be compiled into canonical desired per-port state
B) Port-centric input must be compiled into canonical desired per-port state
C) Dual syntax merge/conflict detection remains as in Step 3/4
D) Step 4 policy overrides remain as in Step 4

Implementation guidance:
- preserve existing input parsing and dual syntax merge where possible
- but ensure the final merged result becomes a canonical desired per-port state, not a backend-ish state

Important:
Do not unnecessarily break existing VlanConfig add/remove/set APIs in this step.
If they still serve as input mutation language, that is acceptable.
But the result of planning should be canonical per-port state.

Acceptance:
- desired runtime state after merge/policy is canonical, not backend-specific

PART 3 — KEEP STEP 4 POLICY LAYER ON CANONICAL STATE

Step 4 logic must continue to operate, but now explicitly on canonical state.

That includes:
- untagged move detection
- delete VLAN in use detection
- mode-none fallback to VLAN 1
- access↔trunk mode-change protection
- dual syntax conflict behavior

Required behavior:
- no Step 4 semantics change
- policy decisions should be applied to canonical desired_per_port
- after policy, canonical desired_per_port must remain valid

Important:
Mode-none fallback to VLAN 1 should still produce canonical state:
- untagged_vlan = 1
- tagged_vlans = empty

Only later should that be compiled to JTCom backend access mode.

Acceptance:
- Step 4 now clearly sits on canonical state, not mixed model state

PART 4 — COMPILE CANONICAL STATE TO JTCom BACKEND ONLY AT FINAL WRITE BOUNDARY

This is the core architectural shift.

Required behavior:
- before calling backend write helpers, canonical desired port state must be converted using the Step 5A conversion helper:
    canonical_to_jtcom_port_vlan_state(...)

This means:
- access canonical state -> JTCom access state
- trunk canonical state -> JTCom native + permit state

Important:
No code before that stage should manually reconstruct:
- permit_vlans
- native_vlan
- access_vlan
except through the shared conversion logic

The project must now have one authoritative canonical -> JTCom compilation path.

Acceptance:
- there is no duplicated ad hoc building of permit/native/access state in planning/apply code
- final write path uses canonical_to_jtcom_port_vlan_state(...)

PART 5 — REMOVE BACKEND LEAKAGE FROM DIFF / PLAN WHERE PRACTICAL

Inspect:
- build_device_plan(...)
- vlan diff helpers
- membership diff rendering
- apply result metadata

Goal:
User-facing diff/plan should reflect canonical truth as much as possible.

That means:
- show untagged/tagged semantics, not raw permit/native semantics,
  unless a view is explicitly backend-facing

Important:
Do not do a giant rewrite of output format if risky.
But ensure current planning data structures are not polluted by backend representation where avoidable.

Acceptance:
- internal planning and comparison are canonical
- backend representation is only used for backend application details

PART 6 — KEEP VERIFICATION STRICT AND CONSISTENT

After Step 5C:
- verification from Step 5B must continue to compare canonical expected vs canonical actual

Required check:
The canonical expected state used for verification must now come from the same canonical desired planning path used for apply.

This is important:
- no secondary reinterpretation layer
- no parallel desired model for verification

Acceptance:
- expected canonical state for verification is derived directly from the same canonical desired plan used for apply

PART 7 — HANDLE UNSUPPORTED CANONICAL STATES CLEANLY AT BACKEND COMPILATION TIME

From Step 5A:
- tagged-only canonical state may be valid canonically
- empty canonical state may be valid canonically
but:
- current JTCom backend compiler may reject some of these directly

Required behavior in Step 5C:
- preserve this separation
- unsupported cases should fail at the backend compilation boundary, not by corrupting canonical planning

Examples:
A) canonical tagged-only port state
   - if JTCom backend cannot represent it, conversion helper raises ValueError
   - that error should be surfaced clearly

B) canonical empty state
   - by Step 4 policy, it should usually already be resolved before backend conversion
   - if it still reaches conversion unexpectedly, fail clearly

Acceptance:
- unsupported JTCom compilation cases are handled as backend-limitation failures, not as canonical-model failures

PART 8 — REUSE STEP 5A / 5B HELPERS, DO NOT DUPLICATE LOGIC AGAIN

You now have:
- port_name_to_id(...)
- get_vlan_membership_type(...)
- validate_canonical_port_state(...)
- canonical_to_jtcom_port_vlan_state(...)
- jtcom_to_canonical_port_vlan_state(...)

Use these helpers.
Do not recreate parallel mini-versions in:
- driver
- policy layer
- diff layer
- apply path

Acceptance:
- one canonical readback conversion path
- one canonical write compilation path
