REQUIRED CHANGES

==================================================
1) MAKE NEW-VLAN MEMBERSHIP INVARIANT EXPLICIT
==================================================

File:
- src/napalm_jtcom/utils/device_diff.py

Current code path:
When a VLAN is being created (vid not in current.vlans), the code computes:

- tagged, untagged = apply_vlan_membership_config(set(), set(), cfg)

and then checks:
- if tagged is None or untagged is None: raise ValueError(...)

This is semantically correct, but the intent is still too implicit.

Why this should be improved:
- For a new VLAN, the baseline is concrete: empty tagged set and empty untagged set
- Therefore the resulting membership must always be concrete
- Returning None here would indicate a programming bug / broken invariant, not a user error

Required improvement:
Replace the generic ValueError-style defensive check with an explicit invariant assertion.

Recommended implementation:
- keep the same logic
- but if tagged is None or untagged is None:
    raise AssertionError(
        f"Internal invariant violated: new VLAN {vid} membership must be concrete"
    )

Alternative acceptable implementation:
- use assert tagged is not None
- use assert untagged is not None
- but prefer a clear explicit AssertionError with a message

Do NOT silently coerce None to empty set.

Expected result:
- code makes it clear that create-time unknown membership is impossible by design
- future regressions fail loudly as internal bugs, not as user-facing misconfiguration

==================================================
2) STRUCTURE UNKNOWN-MEMBERSHIP DIFF METADATA
==================================================

File:
- src/napalm_jtcom/utils/device_diff.py

Current behavior:
When a diff side is touched, but the new result remains unknown (because current baseline was None and desired used add/remove without explicit set), the code adds:

- diff["note"] = "unknown current membership; add/remove target cannot be rendered fully"

This is human-readable, but not ideal for future machine consumers.

Required improvement:
Keep the human-readable note if you want, but add structured metadata so downstream consumers can reason about it without string parsing.

Recommended diff shape:
{
  "from": ...,
  "to": ...,
  "note": "unknown current membership; add/remove target cannot be rendered fully",
  "meta": {
      "unknown_membership": true,
      "reason": "unknown_current_baseline_with_add_remove"
  }
}

At minimum, add:
- diff["meta"] = {"unknown_membership": True}

Preferred:
- include both boolean and normalized reason code

Requirements:
- preserve current semantics
- preserve the existing human-readable message if tests or UX already rely on it
- but add structured metadata alongside it

Do NOT remove the note unless you are certain nothing relies on it.
