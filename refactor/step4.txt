STEP 4 — Introduce explicit policy layer for destructive and ambiguous VLAN membership operations

GOAL

Implement a policy layer on top of the existing canonical VLAN membership system.

This step must formalize behavior for:
- untagged/native VLAN moves
- deleting VLANs that are still in use
- resulting port mode == none
- explicit detach requirements before destructive changes
- unknown baseline behavior (already warning-based)
- preservation of strict dual syntax conflict handling

This step is NOT about redesigning the membership engine.
It is about making destructive and ambiguous operations explicit, safe, and policy-driven.

SELECTED POLICY DECISIONS

These decisions are fixed for this step and must be implemented exactly as follows:

1) Untagged/native move:
   - default: FAIL
   - override flag: allow_untagged_move=True

2) Delete VLAN that is still in use:
   - default: FAIL
   - override flag: allow_vlan_delete_in_use=True

3) Resulting mode == none:
   - automatically map to VLAN 1
   - no allow_mode_none flag
   - mapping must be explicit in diff / warnings / docs

4) Explicit detach:
   - required by default for destructive operations
   - can be bypassed only with explicit override flags

5) Unknown baseline:
   - keep current behavior
   - warning only, not fail

6) Dual syntax conflicts:
   - remain hard FAIL
   - no silent merge and no precedence rules

STRICT NON-GOALS

Do NOT:
- redesign Step 1 canonical model
- redesign Step 2 membership engine
- redesign Step 3 dual syntax merge model
- add new input syntaxes
- weaken existing access↔trunk safety rules
- change None vs [] semantics

This step must extend policy, not replace architecture.

Add the following explicit flags to the main apply/config path:

- allow_untagged_move: bool = False
- allow_vlan_delete_in_use: bool = False

Do NOT add allow_mode_none.
Mode none is handled by automatic VLAN 1 fallback in this step.

These flags must be supported:
- in driver optional_args or equivalent runtime config
- in apply_device_config / set_vlans call path
- in Ansible module and action plugin
- in Galaxy copies
- in docs/examples

Important:
These flags are policy flags and must be clearly documented as potentially dangerous overrides.

HIGH-LEVEL BEHAVIOR

After this step, the flow becomes:

1) Parse and normalize user input
2) Merge dual syntax into canonical VLAN membership
3) Read current state
4) Build canonical per-port membership
5) Run Step 4 policy transformation / validation
6) If validation passes:
   - continue with existing Step 2 membership engine
   - apply
   - verify
7) If validation fails:
   - fail before device mutation

Policy transformation / validation must happen BEFORE:
- backup
- vlan_create
- vlan_delete
- vlan_set_port
- apply_port_changes

PART 1 — UNTAGGED / NATIVE MOVE POLICY

CONTEXT

Currently, assigning a new untagged VLAN to a port overwrites the previous one implicitly.

This is no longer acceptable as default behavior.

Required new behavior:
- default: fail
- only allowed when allow_untagged_move=True

Definition of untagged move:
A port currently has:
- untagged_vlan = A
and desired result would make:
- untagged_vlan = B
where:
- A is not None
- B is not None
- A != B

This includes both:
- VLAN-centric input
- Port-centric input
- native VLAN changes on trunk ports

Required implementation:
1) Add a detection helper, e.g.:
   detect_untagged_move_warnings(current_per_port, desired_per_port) -> list[dict]

Each warning/error entry should contain:
- type: "untagged_move"
- port_id
- current_untagged_vlan
- desired_untagged_vlan
- hint mentioning allow_untagged_move=True

2) Planning behavior:
- if untagged moves are detected and allow_untagged_move is False:
    - check mode: include warning, do not fail
    - apply mode: raise dedicated exception before mutation

Recommended exception:
- VlanMembershipUntaggedMoveError(ValueError)

Message should clearly say:
- port_id
- current VLAN
- desired VLAN
- override hint

3) If allow_untagged_move=True:
- allow the move
- still include warning metadata in check mode and optionally apply result output

Important:
This is separate from access↔trunk mode-change logic.
A pure untagged move from VLAN 20 to VLAN 30 on an access port is NOT the same as access→trunk.

Required tests:
A) port 3 untagged 20 -> desired untagged 30
   - default apply: fail
B) same in check mode:
   - warning only
C) same with allow_untagged_move=True:
   - success
D) same behavior through VLAN-centric and port-centric syntax
E) same behavior for native_vlan change on trunk port

PART 2 — DELETE VLAN IN USE POLICY

CONTEXT

Deleting a VLAN that is still used by ports must no longer be ambiguous.

Selected policy:
- default: fail
- override flag: allow_vlan_delete_in_use=True

Definition of "VLAN in use":
A VLAN is considered in use if any port currently or finally references it as:
- untagged/native VLAN
- tagged VLAN

Required behavior:
1) If a VLAN is marked:
   state: absent
and it is still in use,
then:
- default apply: fail
- check mode: warning
- apply with allow_vlan_delete_in_use=True: allowed

2) What does allow_vlan_delete_in_use=True mean in this step?
It means:
- the system is allowed to detach this VLAN from all affected ports automatically
- then delete it

This auto-detach must be explicit and deterministic.

IMPORTANT:
Do NOT silently auto-detach without the flag.

Required implementation:
A) Add policy planner helper, e.g.:
   detect_vlan_delete_in_use(current_per_port, desired_vlans, desired_per_port) -> list[dict]

Each warning/error entry should include:
- type: "vlan_delete_in_use"
- vlan_id
- affected_ports_tagged
- affected_ports_untagged
- hint mentioning allow_vlan_delete_in_use=True

B) In default apply mode:
- if any VLAN delete-in-use is detected and allow_vlan_delete_in_use=False:
    raise dedicated exception before mutation

Recommended exception:
- VlanDeleteInUseError(ValueError)

C) In check mode:
- warning only

D) If allow_vlan_delete_in_use=True:
- auto-detach this VLAN from all affected ports in the desired state before apply
- then allow normal deletion to continue

Important:
This means the policy layer must modify the effective desired membership plan when allow_vlan_delete_in_use=True.

Recommended implementation approach:
- before passing desired input into membership planning,
  augment the canonical desired_vlans for each absent VLAN:
    - tagged_remove on all affected ports
    - untagged_remove on all affected ports
- then continue through the normal membership engine

This is safer than implementing a special deletion path.

Required tests:
A) VLAN 20 absent, currently untagged on port 3
   - default apply: fail
B) same in check mode:
   - warning
C) same with allow_vlan_delete_in_use=True:
   - auto detach and delete succeeds
D) VLAN 20 absent, currently tagged on ports 5 and 6
   - default fail
E) same with allow_vlan_delete_in_use=True:
   - tagged membership removed and delete proceeds
F) mixed case:
   - native on one port, tagged on another
G) VLAN absent and already unused:
   - allowed without warnings

PART 3 — MODE == NONE POLICY

CONTEXT

A desired result may leave a port with:
- no untagged VLAN
- no tagged VLANs

This is "mode == none".

Selected policy:
- automatically map resulting mode == none to access VLAN 1
- no separate allow flag
- this remapping must be visible in planning output and documentation

Interpretation:
- If the canonical desired membership for a port results in no membership at all,
  the policy layer must transform that port into:
    access VLAN 1
before the backend apply stage.

This is a policy decision, not a backend limitation workaround.

REQUIRED BEHAVIOR

1) Detect ports whose desired final mode is "none"
2) For each such port, transform desired state into:
   - untagged_vlan = 1
   - tagged_vlans = empty set
   - resulting mode = "access"
3) This transformation must happen before low-level apply
4) The transformation must be surfaced to the user in diff / warnings / result metadata

Required warning entry:
- type: "mode_none_mapped_to_vlan1"
- port_id
- mapped_vlan: 1
- reason: "policy_default_vlan1_fallback"

Important:
This must happen consistently for both:
- ports that become none via VLAN-centric input
- ports that become none via port-centric input

Examples:
A) access VLAN 20, then untagged_remove 20 -> becomes access VLAN 1
B) trunk with only tagged VLAN 20 and no native, then tagged_remove 20 -> becomes access VLAN 1

DIFF REQUIREMENTS

If fallback to VLAN 1 happens, the diff must not pretend the result is "none".
It must show the actual effective desired result:
- untagged_vlan: 1
- tagged_vlans: []

Additionally, a warning / metadata note must explain that this came from the Step 4 fallback policy.

IMPLEMENTATION REQUIREMENTS

A) Add helper, e.g.:
   apply_mode_none_fallback(desired_per_port) -> tuple[desired_per_port, warnings]

Behavior:
- scan desired_per_port
- for every port classified as "none":
    set it to access VLAN 1
- return warnings describing which ports were modified by policy

B) This helper must run before:
- backend capability checks
- final desired_port_vlan construction
or must rebuild desired_port_vlan after transformation

C) This fallback must not bypass access↔trunk mode-change protection.
If the fallback causes trunk→access, existing mode-change policy still applies.

This is important:
- mode-none fallback gives the port a valid membership
- but if that implies a dangerous trunk→access transition, existing allow_port_mode_change rules remain in force

Required tests:
A) removing last untagged VLAN on access port
   - desired none -> effective access VLAN 1
B) removing last tagged VLAN on trunk with no native
   - desired none -> effective access VLAN 1
C) diff shows VLAN 1 as final state, not none
D) warning metadata is present
E) if fallback results in trunk→access, mode-change protection still behaves exactly as before
F) no backend unsupported-mode error should occur for these cases anymore, because effective mode is no longer none

PART 4 — EXPLICIT DETACH BY DEFAULT

This is not a separate flag; it is the principle behind the selected policy.

It must be reflected consistently:

- untagged move requires allow_untagged_move
- delete VLAN in use requires allow_vlan_delete_in_use
- mode none is not allowed to remain empty; it is mapped explicitly to VLAN 1

There must be no silent destructive operation in default apply mode except the explicitly chosen VLAN 1 fallback policy.

Review all destructive paths and ensure they align with this principle.
