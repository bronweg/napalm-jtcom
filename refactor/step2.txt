STEP 2 — Implement VLAN membership apply engine with add/remove/set semantics and access↔trunk mode-change protection

GOAL

Implement the runtime logic that applies canonical VLAN membership operations (`add`, `remove`, `set`) to the current switch state.

This step must make VLAN membership behavior deterministic and safe.

After this step, the system must correctly support:

- tagged.add
- tagged.remove
- tagged.set
- untagged.add
- untagged.remove
- untagged.set

with proper conflict handling and explicit policy for access↔trunk transitions.

This is the first step that changes runtime behavior.

HIGH-LEVEL OBJECTIVE

Convert the driver from:
- "implicit replace / ad hoc mutation"

to:
- "explicit membership operations engine"

This step must ensure that:
- trunk permit lists are never accidentally clobbered
- access↔trunk transitions are detected and blocked by default
- membership changes are computed from current state + desired operations
- post-apply verification remains possible and meaningful

STRICT NON-GOALS

Do NOT implement yet:
- dual syntax from port-centric input
- untagged conflict policy flags like fail/move
- VLAN delete-in-use policies
- action plugin schema redesign
- collection-level docs overhaul

This step is about the internal membership engine and safe runtime semantics only.

FILES TO MODIFY

Primary:
- src/napalm_jtcom/driver.py

New helper module(s) strongly recommended:
- src/napalm_jtcom/utils/vlan_membership.py
OR
- src/napalm_jtcom/logic/vlan_membership.py

Tests:
- tests/unit/test_vlan_membership_apply.py
- optionally expand existing driver tests if present

Optional:
- small adjustments to action plugin / module only to pass a new boolean flag:
    allow_port_mode_change
  but do not redesign the public API beyond that

REQUIRED OUTCOME

The driver must be able to:

1) Read current membership
2) Compute desired membership after applying add/remove/set
3) Detect whether any port changes mode:
   - access -> trunk/hybrid
   - trunk/hybrid -> access
4) Fail by default if such a mode change is required
5) Allow it only when allow_port_mode_change=True
6) Apply the full computed per-port desired state safely
7) Verify the result after applying

SEMANTIC RULES TO IMPLEMENT

==================================================
1) CANONICAL CURRENT STATE REPRESENTATION
==================================================

Build a canonical in-memory representation of current VLAN membership per port.

Required shape:

current_per_port = {
    port_id: {
        "untagged_vlan": int | None,
        "tagged_vlans": set[int],
    },
    ...
}

Important:
- internal port_id remains 0-based
- tagged_vlans must be a set
- untagged_vlan must be a single VLAN or None

This structure is the runtime source of truth for Step 2.

==================================================
2) HOW TO BUILD current_per_port
==================================================

Use existing read paths already present in the driver.

Preferred source order:
1. If you already have reliable data from both:
   - vlan static page
   - port-based VLAN page
   then use the most accurate combination
2. If not, build best-effort from existing VLAN-centric read logic

Requirements:
- For every known port, initialize:
    {"untagged_vlan": None, "tagged_vlans": set()}
- Then populate membership from current VLAN data

Rules:
- If a port is listed as untagged in VLAN X:
    current_per_port[port]["untagged_vlan"] = X
- If a port is listed as tagged in VLAN X:
    current_per_port[port]["tagged_vlans"].add(X)

If current read reveals impossible state:
- same port untagged in more than one VLAN
then:
- raise a clear exception
- do not try to “fix” silently

==================================================
3) APPLY DESIRED VlanConfig OPERATIONS TO CURRENT STATE
==================================================

Input:
- current_per_port
- desired VLAN patch objects (VlanConfig instances)
- each VlanConfig uses normalized_membership()

Build:
- desired_per_port

Algorithm:
- start with a deep copy of current_per_port
- apply each VLAN patch one by one
- all changes must be computed in memory first
- do NOT talk to device during this phase

For each VLAN cfg:
- membership = cfg.normalized_membership()

3A) Tagged side

If membership["tagged"]["set"] is not None:
- For every port in desired_per_port:
    - remove cfg.vlan_id from tagged_vlans
- For every port in membership["tagged"]["set"]:
    - add cfg.vlan_id to tagged_vlans

Else:
- For every port in membership["tagged"]["add"]:
    - add cfg.vlan_id to tagged_vlans
- For every port in membership["tagged"]["remove"]:
    - remove cfg.vlan_id from tagged_vlans if present

3B) Untagged side

If membership["untagged"]["set"] is not None:
- For every port whose current untagged_vlan == cfg.vlan_id:
    - set untagged_vlan = None
- For every port in membership["untagged"]["set"]:
    - set untagged_vlan = cfg.vlan_id

Else:
- For every port in membership["untagged"]["add"]:
    - set untagged_vlan = cfg.vlan_id
- For every port in membership["untagged"]["remove"]:
    - if current untagged_vlan == cfg.vlan_id:
        set untagged_vlan = None

IMPORTANT:
At this stage, untagged assignment may overwrite an existing untagged_vlan from another VLAN.
That is acceptable for now.
Do NOT introduce separate fail/move policy yet.
Just make the behavior deterministic and documented.

==================================================
4) ENSURE TAGGED/UNTAGGED CONSISTENCY
==================================================

After applying all desired operations in memory:

For each port:
- if untagged_vlan is not None
- and untagged_vlan exists in tagged_vlans
then:
- remove that VLAN from tagged_vlans

Rationale:
A VLAN cannot be both tagged and untagged on the same port.

This must be enforced automatically in desired_per_port.

==================================================
5) PORT MODE CLASSIFICATION
==================================================

Define a helper:

classify_port_mode(port_state) -> str

Rules:
- if tagged_vlans is non-empty:
    return "trunk"
- elif untagged_vlan is not None:
    return "access"
- else:
    return "none"

For this step:
- treat any presence of tagged VLANs as trunk/hybrid
- no need to distinguish trunk vs hybrid yet

==================================================
6) ACCESS↔TRUNK MODE CHANGE POLICY
==================================================

This is mandatory.

For each port:
- current_mode = classify_port_mode(current_per_port[port])
- desired_mode = classify_port_mode(desired_per_port[port])

A mode change is considered dangerous if:
- current_mode == "access" and desired_mode == "trunk"
- current_mode == "trunk" and desired_mode == "access"

Default behavior:
- fail

Add support for:
- allow_port_mode_change: bool = False

Behavior:
- if a dangerous mode change is detected and allow_port_mode_change is False:
    - in normal apply mode: raise a clear exception
    - in check mode: do NOT raise; instead record a warning in result metadata

Required error/warning contents:
- port_id
- current_mode
- desired_mode
- hint to set allow_port_mode_change=true

Important:
Do NOT block:
- access -> none
- none -> access
- none -> trunk
unless you explicitly want to, but for this step only access↔trunk transitions are protected.

==================================================
7) COMPUTE FINAL DEVICE-LEVEL INTENT PER PORT
==================================================

Before sending anything to the switch, compute a final desired port VLAN view:

desired_port_vlan = {
    port_id: {
        "mode": "access" | "trunk" | "none",
        "native_vlan": int | None,
        "permit_vlans": list[int],
    }
}

Rules:
- mode:
    - "trunk" if tagged_vlans non-empty
    - "access" if tagged_vlans empty and untagged_vlan is not None
    - "none" otherwise

- native_vlan:
    - untagged_vlan or None

- permit_vlans:
    - if mode == "trunk":
        sorted(tagged_vlans union {native_vlan if native_vlan is not None})
    - if mode == "access":
        [untagged_vlan]
    - if mode == "none":
        []

IMPORTANT:
This is critical:
NEVER build permit_vlans from only the delta.
Always build the full desired permit list for the port.

This is how we avoid clobbering trunk state accidentally.

==================================================
8) TRANSLATE desired_port_vlan TO DEVICE CALLS
==================================================

Use the existing switch-specific driver functions/endpoints.

Requirements:
- apply per-port final desired state
- do not send partial trunk payloads that can erase existing VLANs unintentionally
- always send full desired port state for each changed port

Implementation strategy:
- compare current_per_port vs desired_per_port
- derive changed ports only
- for each changed port:
    - compute desired mode/native/permit
    - call the device update function with full desired state

If your current low-level API only supports VLAN-centric mutation:
- refactor it so that the final write path is port-centric internally
- but do not expose port-centric user syntax yet

==================================================
9) CHECK MODE BEHAVIOR
==================================================

Support a dry-run path.

In check mode:
- compute desired_per_port
- compute changed ports
- detect mode-change warnings
- do not touch device
- return deterministic diff/plan metadata

If mode change would fail in apply mode:
- do NOT raise in check mode
- instead include warning(s) in result

Recommended result shape for check mode:
{
    "changed": bool,
    "warnings": [...],
    "changed_ports": [...],
    "changed_vlans": [...],
    "before": ...,
    "after": ...,
}

==================================================
10) POST-APPLY VERIFICATION
==================================================

After real apply:
- re-read current membership from the device
- rebuild final per-port state
- compare to desired_per_port

If mismatch remains:
- raise a clear verification error
- include remaining diff

Do not silently succeed if verification fails.

11) TESTS — PURE LOGIC
==================================================

Create:
- tests/unit/test_vlan_membership_apply.py

These tests should cover the pure membership engine without real device calls.

Required test cases:

A) tagged.add merges
current:
    port 5 tagged {10,30}
desired:
    VLAN 20 tagged.add [5]
result:
    port 5 tagged {10,20,30}

B) tagged.remove subtracts
current:
    port 5 tagged {10,20,30}
desired:
    VLAN 20 tagged.remove [5]
result:
    port 5 tagged {10,30}

C) tagged.set replaces
current:
    port 5 tagged {10,20,30}
desired:
    VLAN 40 tagged.set [5]
result:
    port 5 tagged contains 40 and no longer contains 10/20/30 for that VLAN's dimension

D) untagged.add assigns native VLAN
current:
    port 3 untagged 10
desired:
    VLAN 20 untagged.add [3]
result:
    port 3 untagged 20

E) untagged.remove clears native if matching
current:
    port 3 untagged 20
desired:
    VLAN 20 untagged.remove [3]
result:
    port 3 untagged None

F) tagged/untagged same VLAN on same port is auto-normalized
current:
    port 3 tagged {20}
desired:
    VLAN 20 untagged.add [3]
result:
    port 3 untagged 20 and VLAN 20 removed from tagged_vlans

G) access -> trunk fails by default
current:
    port 3 untagged 20, tagged empty
desired:
    add tagged VLAN 30 to port 3
result:
    exception unless allow_port_mode_change=True

H) trunk -> access fails by default
current:
    port 5 untagged 10, tagged {20,30}
desired:
    remove all tagged VLANs, leave only untagged
result:
    exception unless allow_port_mode_change=True

I) access -> trunk allowed with flag
same as G, with allow_port_mode_change=True
result:
    success

J) check mode does not fail on mode change
same as G, check_mode=True
result:
    warning present, no exception

K) none vs [] semantics remain intact indirectly
ensure omitted legacy fields do not produce replacement behavior accidentally
